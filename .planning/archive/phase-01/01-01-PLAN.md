# Plan 01-01: Bridge Capability Contracts

## Status: RETROACTIVE VALIDATION

This plan describes the **ideal** implementation of capability contracts in the bridge.
Commit `5d1cde9` on the bridge already implements this plan. Execution consists of
**verifying** the existing code matches this spec, adding any missing real-world fixtures
for TEST-02, and confirming the CI gate (TEST-03).

## Goal

All six CAPA requirement types live in `src/capability/` with Zod schemas as source of truth,
TS types derived via `z.infer`, and comprehensive tests that validate both valid and invalid
fixtures. The barrel export at `@x9-forge/contracts/capability` is the single import path.

## must_haves

1. `CapabilityManifest` resolves `serviceName?` divergence between X9 (absent) and Forge (optional)
2. `CapabilityRegistryEntry` canonical shape = `{ name, enabled, host, port, version, protocol?, tools? }` with `toEndpoint()` / `fromEndpoint()` helpers
3. `ToolCallResponse` is a Zod discriminated union on `status` field
4. Every schema has valid-parse + reject-invalid tests with realistic fixtures
5. CI blocks merge if any contract test fails

## Threat Model

<threat_model>
### Attack Surface
- Zod schemas in `src/capability/` define the validation boundary for all cross-repo payloads.
- If a schema is too permissive, invalid data crosses the bridge undetected (P-02 pitfall).
- If a schema is too strict, legitimate payloads are rejected causing service outages.

### ASVS L1 Threats Addressed
- **V5.1 Input Validation**: Every schema enforces `min(1)` on string fields (no empty strings),
  `positive()` on port numbers, enum constraints on status fields. This prevents injection of
  empty/malformed data at the contract boundary.
- **V5.2 Sanitization**: `z.record(z.string(), z.unknown())` for `inputSchema` and `input` fields
  accepts arbitrary JSON but constrains the container shape. No HTML/SQL sanitization needed
  (these are machine-to-machine contracts, not user input).

### Mitigations
1. **Schema = type derivation**: `type X = z.infer<typeof XSchema>` pattern prevents P-03 (schema/type drift).
   No manually declared types alongside Zod schemas.
2. **Fail-loud parse**: Tests verify `safeParse` rejects invalid fixtures. Consumers MUST call
   `.parse()` at boundaries, not just rely on TS types.
3. **Real fixtures**: TEST-02 uses fixtures derived from actual service manifests, not invented data.
   Catches shape assumptions that pass synthetic tests but fail on production payloads.
4. **No credential data in schemas**: `credentials` in `ToolCallRequest` is `Record<string, string>` —
   the bridge never inspects credential values, only validates the container shape. Credential
   content is opaque (encrypted at rest in vault, decrypted only at dispatch time).
</threat_model>
