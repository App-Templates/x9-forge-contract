# Plan 01-02: X9 Migration â€” Re-export Shim + Registry Canonical

## Status: RETROACTIVE VALIDATION

X9 migration was partially implemented outside GSD tracking. The compat shims in
`packages/types/src/capability.ts`, `health.ts`, `env-schema.ts` already re-export from
bridge. `scripts/generate-registry.ts` already uses `fromEndpoint()`. `registry.ts` already
uses `CapabilityRegistryEntrySchema` + `toEndpoint()` with legacy backward compat.

This plan describes the ideal state and verifies the existing code matches it.

## Goal

X9 `packages/types/` becomes a pure compat shim that re-exports all capability types from
`@x9-forge/contracts/capability`. Zero local type definitions for contracts that live in the
bridge. `generate-registry.ts` produces canonical `{ host, port, version }` entries.
`registry.ts` accepts both canonical and legacy formats for backward compatibility with
existing VPS `registry.json` files. All X9 services compile and pass tests without changes
to their import paths.

## must_haves

1. `packages/types/src/capability.ts` re-exports all types + schemas from `@x9-forge/contracts/capability`
2. `packages/types/src/health.ts` re-exports `HealthStatus` + `HealthStatusSchema` from bridge
3. `packages/types/src/env-schema.ts` re-exports `EnvSchemaField`, `EnvSchemaDoc` + schemas from bridge
4. `generate-registry.ts` uses `fromEndpoint()` to produce canonical entries
5. `registry.ts` uses `z.union([CanonicalEntrySchema, LegacyEntrySchema])` for backward compat
6. Zero import path changes in X9 services (shim is transparent)
7. X9 build + existing tests pass post-migration

## Threat Model

<threat_model>
### Attack Surface
- The compat shim in `packages/types/` is a re-export layer. If it accidentally re-exports
  a type with a different name or shape, X9 services would use a mismatched contract.
- `registry.ts` backward-compat `z.union` could accept a malformed legacy entry if the
  legacy schema is too permissive.
- `generate-registry.ts` uses `fromEndpoint()` to parse URLs â€” malformed manifest endpoints
  could produce invalid registry entries.

### ASVS L1 Threats Addressed
- **V5.1 Input Validation**: `registry.ts` validates ALL registry entries through Zod schemas
  at load time. Legacy entries are validated by `LegacyEntrySchema`, canonical by
  `CapabilityRegistryEntrySchema`. No unvalidated JSON enters the registry.
- **V1.14 Configuration**: Bridge dependency is pinned by SHA (not semver range). This prevents
  supply-chain drift where a bridge update silently changes the contract.

### Mitigations
1. **Pure re-export shim**: `capability.ts` contains ONLY `export type/export` from bridge.
   No local type definitions that could drift from the bridge source of truth (P-03 mitigation).
2. **No circular dependency**: Shim direction is consumer -> bridge only. Bridge never imports
   from X9. ESLint `no-restricted-imports` in bridge prevents reverse dependency (P-01 mitigation).
3. **Legacy backward compat**: `z.union` with `LegacyEntrySchema` ensures existing VPS deployments
   continue working. The legacy schema is strict (requires `endpoint` as non-empty string).
4. **URL parsing safety**: `fromEndpoint()` uses `new URL()` which throws `TypeError` on invalid
   URLs. This is fail-loud: a malformed manifest endpoint crashes `generate-registry.ts` at
   build time, not at runtime.
</threat_model>
