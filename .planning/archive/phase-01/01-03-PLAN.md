# Plan 01-03: Forge v2 Migration — Re-export Shim + Factory Update + Contract Test

## Status: NEW WORK (not yet implemented)

Forge `packages/types/src/x9.ts` still defines local interfaces. This plan migrates them
to re-export from the bridge, replacing `X9CapabilityManifest`, `X9CapabilityRegistryEntry`,
`X9EnvSchemaField`, and `X9EnvSchemaDoc` with aliased re-exports from
`@x9-forge/contracts/capability`.

`X9AgentContext` stays local in `x9.ts` — it is Phase 2 scope.

## Goal

Forge `packages/types/src/x9.ts` becomes a compat shim that re-exports capability types
from the bridge with backward-compatible aliases (e.g., `CapabilityManifest as X9CapabilityManifest`).
All Forge services compile without changing their import paths. `deploy.machine.ts` produces
structurally valid `CapabilityRegistryEntry` objects. `x9.client.ts` types align with bridge
shapes. Forge tests pass and staging deployment is verified.

## must_haves

1. `x9.ts` re-exports capability types from bridge with `X9` prefix aliases for backward compat
2. `X9AgentContext` remains as local interface in `x9.ts` (Phase 2 scope, not touched)
3. `@x9-forge/contracts` added as dependency to Forge (git+https with SHA or pnpm override)
4. All Forge services compile without import path changes
5. `deploy.machine.ts` output is structurally valid against `CapabilityRegistryEntrySchema`
6. Forge test suite passes (229+ tests)

## Threat Model

<threat_model>
### Attack Surface
- Aliased re-exports (`CapabilityManifest as X9CapabilityManifest`) create a naming indirection.
  If the bridge type structurally diverges from what Forge services expect, the `as` cast at
  runtime (`res.json() as X9CapabilityManifest` in `x9.client.ts:55`) could silently accept
  invalid payloads.
- Adding `protocol?` and `tools?` optional fields to `X9CapabilityRegistryEntry` via bridge
  widening — these are additive and safe, but Forge code that spreads `...entry` into other
  objects could inadvertently pass unexpected fields.
- Bridge dependency via `git+https` uses HTTPS token auth. On CI, the `GITHUB_TOKEN` must have
  read access to the private bridge repo.

### ASVS L1 Threats Addressed
- **V5.1 Input Validation**: Forge `x9.client.ts` casts `res.json() as X9CapabilityManifest`
  without runtime validation. This is pre-existing (not introduced by migration). Phase 4
  will add `CapabilityManifestSchema.parse()` at the fetch boundary. For Phase 1, the
  structural type alignment ensures compile-time safety.
- **V1.14 Configuration**: Bridge SHA pinning ensures Forge cannot accidentally pick up a
  breaking bridge change. The `pnpm.overrides` link pattern for local dev is excluded from
  CI (CI uses the git+https SHA).
- **V14.2 Dependency**: The bridge has zero runtime dependencies (only `zod` as peerDep).
  Adding it to Forge does not expand the attack surface beyond Zod, which Forge already uses.

### Mitigations
1. **Alias transparency**: `export type { A as B }` is a TypeScript-only operation. At runtime,
   the imported value is identical to the bridge export. No wrapper, no transformation, no
   opportunity for data corruption.
2. **Optional field safety**: Bridge adds `protocol?` and `tools?` to `CapabilityRegistryEntry`.
   Forge code that constructs entries without these fields continues to compile because they
   are optional. Forge code that reads entries can ignore them safely (they are `undefined`
   when absent, which is the same as before migration).
3. **No credential exposure**: The bridge dependency contains zero secrets. It is a pure type
   package. The `git+https` URL uses GitHub token for repo access, not for runtime secrets.
4. **`X9AgentContext` isolation**: By keeping `X9AgentContext` as a local interface (Phase 2
   scope), we avoid prematurely migrating a complex type that includes `credentials: Record<string, string>`.
   The credential type will be migrated as `AgentCredentials` discriminated in Phase 2 with
   proper migration path and VPS context.json compatibility verification.
</threat_model>
