# Plan 02-01: Research — VPS context.json Inventory

## Goal

Inventory all `context.json` files on the production VPS to detect edge cases before
writing any schema code. Extract a representative fixture for `parseAgentContext` tests.
This plan requires operator (Stefano) SSH access to VPS.

## Tasks

<task id="02-01-01">
<title>Inventory VPS context.json files — field shapes and edge cases</title>
<read_first>
- .planning/phases/02-agentcontext-split-block-b/02-RESEARCH.md
</read_first>
<action>
SSH into the VPS and run the following command to extract all context.json files:

```bash
find /data/agents -name context.json -exec echo "=== {} ===" \; -exec cat {} \;
```

For each file, verify these fields exist and record their types:
- `agentId` (string, non-empty)
- `ownerId` (string, non-empty)
- `workspacePath` (string, absolute path)
- `registryPath` (string, absolute path)
- `credentials` (object with string values)
- `llmConfig` (object with `provider` and `model`)
- `telegramBotToken` (string)
- `telegramAllowFrom` (array of strings)
- `displayName` (string)

Detect edge cases:
1. Are there agents with empty `credentials` (`{}`)?
2. Are there agents with `telegramAllowFrom: ['*']` (legacy wildcard)?
3. Are there agents missing `displayName` (added later)?
4. Are there any unexpected extra fields not in the known schema?
5. What are the actual credential keys used across all agents?

Record findings in the sidecar `tests/agent/fixtures/COMPAT-NOTES.md` file.
</action>
<acceptance_criteria>
- A list of all agent IDs found on VPS is documented
- Each unique field shape variant is documented
- Edge cases (missing fields, unexpected fields, empty values) are listed
- Total number of context.json files is recorded
- `grep -c 'agentId' tests/agent/fixtures/context-production-sample.json` returns >= 1
- `grep -c 'ownerId' tests/agent/fixtures/context-production-sample.json` returns >= 1
- `grep -c 'credentials' tests/agent/fixtures/context-production-sample.json` returns >= 1
</acceptance_criteria>

echo "Manual SSH step — no automated verification"
</automated>
</task>
