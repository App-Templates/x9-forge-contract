# Plan 04.1-01: Bridge — NoAuthBridgeClient

## Goal

Add `NoAuthBridgeClient` interface + widen `createBridgeClient<A extends 'secret' | 'token' | 'none'>` so the factory produces a typed no-auth client for the 3 capability discovery endpoints (`GET /:cap/manifest`, `GET /:cap/env-schema`, `GET /:cap/health`). Closes R-09 surfaced during Phase 4 — Forge currently has a local `capBridgeClient` helper duplicating bridge plumbing because the factory rejected `'none'` auth.

This plan is **chirurgico**: widen one type parameter, add one interface, add one factory branch, add unit tests. Do NOT refactor the existing `'secret'` / `'token'` branches. Do NOT modify endpoint contract files in `src/http/endpoints/` (cap-manifest/cap-env-schema/cap-health were already shipped with `authType: 'none'` in 04-01).

<threat_model>
## Security Threat Model (ASVS Level 1)

| ID | Threat | Severity | Mitigation | Status |
|----|--------|----------|------------|--------|
| T-4.1-01 | NoAuth client used for endpoint requiring auth | LOW | `BridgeClient<A>` conditional type narrows return to `NoAuthBridgeClient` for `'none'` — exposes only `capManifest`/`capEnvSchema`/`capHealth`; calling `listAgents()` is TS2339 | MITIGATED by conditional type |
| T-4.1-02 | Caller passes auth headers via `options.headers` to noauth client | LOW | WR-02 fix (commit `fd8074a`) already swapped spread order; `options.headers` cannot override `auth` | MITIGATED in Phase 4 |
| T-4.1-03 | NoAuth client widens factory contract → existing secret/token clients accidentally accept `none` auth at compile-time | MEDIUM | New generic constraint stays exclusive — `'secret' | 'token' | 'none'` is a string union, not a structural widening; existing call sites still bind to `<'secret'>` / `<'token'>` explicitly | MITIGATED by generic discipline + tests |

**Blocking threats:** None. T-4.1-03 is the only design-sensitive one — task 04.1-01-02 adds a compile-time test that ensures secret/token clients don't gain noauth methods.
</threat_model>

<actions>

   - Add `NoAuthBridgeClient extends BaseBridgeClient<'none'>` interface (after `TokenBridgeClient` declaration ~L111). Methods:

     - `capEnvSchema(signal?: AbortSignal): Promise<CapEnvSchemaResponse>`

     - **No `cap` arg.** Capability services mount manifest/env-schema/health at root (`http://cap-voice:3000/manifest`); the cap NAME is implicit in the `baseUrl`.    - **Defect surfaced during planning:** Phase 4 contracts at `src/http/endpoints/cap-manifest.ts:15`, `cap-env-schema.ts:14`, `cap-health.ts:14` declare `path: '/:cap/manifest' as const` (etc.) — this does NOT match the real wire pattern. Fix in this same task: update each contract's `path` string to `'/manifest'` / `'/env-schema'` / `'/health'`. JSDoc above each (`* GET /:cap/manifest`) should also drop the `:cap` prefix. 1-line change per file. The path field is documentation-only at runtime (typed methods construct their own paths) so this is a safe documentation correction.
   - Widen `BridgeClient<A>` conditional type (~L121) to add `A extends 'none' ? NoAuthBridgeClient : never` branch
   - Widen `createBridgeClient` factory generic to `<A extends 'secret' | 'token' | 'none'>` (~L159)
   - Add `'none'` branch in factory body: `else if (authType === 'none' || Object.keys(auth).length === 0)` returning a `noAuthClient: NoAuthBridgeClient`. Mirror the secret/token client structure: `request<T>` reused, `authType` getter, typed methods that call `*RequestSchema.parse(input)` for path params and `*ResponseSchema.parse(raw)` on outputs (matching WR-01 fix pattern from `e3c1173`).
   - Note: since `AuthNone` is `Record<string, never>` (or empty object type — verify), the discriminator `'X-Internal-Secret' in auth` already excludes noauth; add an explicit `else` branch.
2. Smallest viable diff. Do NOT reformat existing code. Do NOT touch the secret/token branches.
</actions>
