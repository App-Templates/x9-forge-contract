# Phase 04.1 Code Review — NoAuthBridgeClient + capBridgeClient consolidation

## Summary

Implementation of `NoAuthBridgeClient` is sound. The `createBridgeClient<A extends 'secret' | 'token' | 'none'>` widening cleanly preserves the discriminated-return-type discipline, the runtime-parse discipline (WR-01 from Phase 4) is consistently applied in all three new methods, and the `@ts-expect-error` compile-time guards do effectively pin down the intended type constraints (I verified: each directive is placed on a property access that TypeScript actually rejects at the current definition, so relaxing the type in either direction would flip them into "unused directive" errors).

No critical issues. Findings are all about stale documentation that still references the old `/:cap/*` path convention (the very typo this phase fixed) and a handful of minor test hygiene improvements.

### WR-01: Stale `/:cap/*` docstring in `AuthNone`
**Severity:** warning
**File:** /Users/admintemp/Downloads/Claude/x9-forge-contract-bridge/src/auth/auth-headers.ts:38-40
**Issue:** The JSDoc for `AuthNone` still says "Used by capability discovery endpoints: GET /:cap/manifest, GET /:cap/env-schema, GET /:cap/health." This is precisely the convention that Phase 04.1 retired — capability identity is in the `baseUrl` host, not a path prefix. The `path` strings on the three contract files were corrected, but this docstring was missed and now misdirects readers.
**Recommendation:** Update the JSDoc block to read something like "Used by capability discovery endpoints: `GET /manifest`, `GET /env-schema`, `GET /health` (capability identity is conveyed by the caller's baseUrl, not a path prefix)."

### IN-01: `mockFetchOnce` reuses a single Response instance
**Severity:** info
**File:** /Users/admintemp/Downloads/Claude/x9-forge-contract-bridge/tests/http/no-auth-client.test.ts:49-58
**Issue:** `mockFetchOnce` constructs the `Response` once and returns the same reference from every `fetch()` invocation. `Response.json()` / the body stream can only be consumed once, so if a future test mistakenly triggers a second fetch (retry loop, etc.) it will throw `TypeError: Body has already been consumed` rather than a clear assertion failure. The current tests each trigger one fetch so the bug is latent, but the helper is fragile by design.
**Recommendation:** Build a fresh `Response` per call, e.g. `vi.fn(async () => new Response(JSON.stringify(body), { status, headers: ... }))`, matching the style already used in `captureFetch`.

### IN-03: `BridgeClient` default type parameter is the full union
**Severity:** info
**File:** /Users/admintemp/Downloads/Claude/x9-forge-contract-bridge/src/http/bridge-client.ts:141-145
**Issue:** `BridgeClient<A = 'secret' | 'token' | 'none'>` defaults to the full union, which collapses to `never` under the conditional chain. Callers who rely on inference (e.g., `const c = createBridgeClient({ baseUrl, auth: { 'X-Internal-Secret': 's' } })`) get the right narrowed type because inference on `A` succeeds from the `auth` constraint — but anyone who *explicitly* writes `BridgeClient` without a generic parameter gets `never`, which is surprising. Phase 4's original type had the same property for the `'secret' | 'token'` union. Consider tightening or documenting.
**Recommendation:** Either drop the default (force an explicit generic) or change the default to `'secret'` to match the historical primary use case. Low priority — no regression.

### IN-04: Runtime auth discriminator is compile-time-only safe
**Severity:** info
**File:** /Users/admintemp/Downloads/Claude/x9-forge-contract-bridge/src/http/bridge-client.ts:227-229
**Issue:** The runtime branch selector uses `'X-Internal-Secret' in auth` / `'X-Internal-Token' in auth`. If a caller bypasses TypeScript (e.g., `any`, unsafe cast, JS consumer) and passes `{ 'X-Internal-Secret': 'x' }` together with `<'none'>`, they get silently routed into the secret branch and the noauth methods are absent from the returned object. The compile-time guarantee holds; the runtime has no defence-in-depth check. This matches the existing secret/token branch behaviour, so it is not a regression.
**Recommendation:** Optional: add a cheap runtime assertion — when `A === 'none'`, `Object.keys(auth).length === 0`, otherwise throw a clear `Error`. Keeps the JS/edge-caller story honest. Not required.

### IN-05: `RequestInit.headers` narrowing in tests is loose
**Severity:** info
**File:** /Users/admintemp/Downloads/Claude/x9-forge-contract-bridge/tests/http/no-auth-client.test.ts:137-140
**Issue:** The test casts `init?.headers as Record<string, string> | undefined` and then indexes into it. `RequestInit.headers` is `HeadersInit` which is a union of `Headers`, `Record<string, string>`, and `[string, string][]`. The current `request()` implementation always passes a plain object so this works, but a future refactor (e.g., using `Headers` instance) would make the assertion silently pass even if a secret header sneaked in.
**Recommendation:** Normalise via `new Headers(init?.headers).get('X-Internal-Secret')` so the test is robust to any `HeadersInit` shape. Cosmetic.
