# Plan 04-01: Bridge — 11 Endpoint Contracts + createBridgeClient Full + Unit Tests + Real Fixtures

## Goal

Define Zod-backed request/response/auth contracts for all 11 cross-repo HTTP endpoints
in the bridge. Extend `createBridgeClient` with typed endpoint methods that enforce
correct auth, request body, and response shape at compile-time. Each endpoint has
real-fixture-based unit tests that validate Zod parse against actual production shapes
derived from agent-x9 and forge-v2 source code.

This plan delivers the core bridge-side contracts. Plans 04-03 and 04-04 consume these
contracts in the X9 and Forge repos respectively.

**Response shape standardization (HTTP-13, HTTP-14):** The current X9 agent-core endpoints
return non-standard shapes (e.g., `{ agents: [...] }`, `{ ok: true, agentId }`,
`{ answer: string }`). The bridge contracts type the REAL shapes as they exist today
AND export the standard `BridgeSuccessResponse<T>` / `BridgeErrorResponse` shapes.
Migration to standardized shapes happens incrementally in 04-03 (X9) and 04-04 (Forge)
at the boundary layer. The bridge contracts support BOTH shapes during transition.

<threat_model>
## Security Threat Model (ASVS Level 1)

| ID | Threat | Severity | Mitigation | Status |
|----|--------|----------|------------|--------|
| T-4-01 | Wrong auth type sent to endpoint (Bug #15 class) | HIGH | Each endpoint contract declares `authType: 'secret' | 'token' | 'none'`; `createBridgeClient` typed methods enforce correct auth at compile-time via `AuthForEndpoint<T>` | MITIGATED by AUTH-03 + AUTH-04 (Phase 3) + endpoint declarations (this plan) |
| T-4-02 | Path parameter injection in `/internal/agents/:agentId/*` | HIGH | Zod regex validation `z.string().regex(/^[a-z0-9-]+$/)` on agentId path param in request schemas; matches agent-core's `AGENT_ID_SCHEMA` | MITIGATED by schema validation |
| T-4-03 | SSE frame injection / malformed JSON in stream frames | MEDIUM | SSE frame schemas validate each frame type discriminated on `type` field; parser rejects unknown frame types | MITIGATED in plan 04-02 (SSE frames) |
| T-4-04 | Response body confusion: error body parsed as success | MEDIUM | `ok` literal discriminator (`true`/`false`) in `BridgeResponseSchema` prevents confusion; endpoint-specific schemas narrow the `data` field | MITIGATED by HTTP-13/HTTP-14 |
| T-4-05 | Credential leakage in request/response shapes | MEDIUM | No credential values in typed request/response shapes; auth headers carry secrets but are typed as `string` (opaque); `PostCallPayload` does not expose per-agent vault secrets | MITIGATED by design |
| T-4-06 | Unbounded array/string in request bodies | LOW | Zod schemas use `.min(1)` on required strings and `.max()` constraints where applicable; history array has structural validation per element | MITIGATED by schema constraints |

**Blocking threats:** T-4-01 (HIGH) and T-4-02 (HIGH) — both mitigated by this plan's deliverables.
</threat_model>

/**

  TMethod extends 'GET' | 'POST',
  TPath extends string,
  TAuth extends EndpointAuthType,

  readonly paramsSchema?: TParams;
  readonly bodySchema?: TBody;
  readonly responseSchema: TResponse;
}
